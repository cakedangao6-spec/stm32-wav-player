# STM32 W25Q64 读 WAV + PWM 音频播放学习步骤

## 学习目标

在已掌握硬件 SPI 读写 W25Q64 的基础上（11-2），逐步实现一个完整的音频播放链路：从 W25Q64 Flash 读取 WAV 文件、解析 WAV 头获取 PCM 参数、用 PWM 方式把数字音频变成声音。学习顺序坚持从简单到复杂：先把 W25Q64 读取回顾清楚，再把 WAV 文件格式搞清楚，然后理解 PWM 发声原理，引入定时器按采样率输出，最后用双缓冲保证播放连续。

## 学习习惯与协作规则

- 每进入一个阶段，先把阶段拆成明确小步，并说明总共有几步、当前进行到第几步。

## 代码风格规范

### 文件组织
- **自写的 .c/.h** → 统一放在 `Hardware/` 目录下
- **主函数 main.c/main.h** → 留在 `User/` 目录下
- **标准外设库** → 放在 `Library/` 目录，不要修改
- **启动/内核文件** → 放在 `Start/` 目录，不要修改
- 每个硬件模块一个 .c 和一个 .h，不要在 .c 里放不相关的函数

### 注释格式
- **文件头注释**：用 `@file` + `@brief` 说明文件名和功能
  ```c
  /**
    * @file    xxx.c
    * @brief   一句话描述文件功能
    */
  ```
- **函数注释**：函数前用 Doxygen 风格，中文说明
  ```c
  /**
    * 函    数：函数名（用中文描述功能）
    * 参    数：参数名 参数说明
    * 返 回 值：返回值说明
    * 说    明：补充说明（可选）
    */
  ```
  - `函    数：`、`参    数：`、`返 回 值：` 对齐
  - 参数和返回值中没有合适的内容时写"无"
- **行内注释**：`//` 放在代码右侧，解释"为什么这样做"，不解释"做了什么"
- **宏/常量注释**：定义处用行内注释说明用途

### 命名规范
| 类别 | 风格 | 示例 |
|------|------|------|
| 函数 | PascalCase，模块名_功能名 | `RingBuffer_Init`、`W25Q64_ReadID`、`USART1_SendString` |
| 变量 | snake_case | `rx_ring`、`dma_write_index`、`line_buffer` |
| 宏/常量 | UPPER_CASE | `RING_BUF_SIZE`、`W25Q64_PAGE_SIZE`、`LED_PIN` |
| 类型定义 | PascalCase + `_t` 后缀 | `RingBuffer_t`、`WavInfo_t`、`AudioPlayerState_t` |
| 枚举值 | UPPER_CASE | `AUDIO_PLAYER_IDLE`、`WAV_STATUS_OK` |

### 缩进与排版
- **缩进**：统一使用 4 个空格（不用 Tab）
- **大括号**：K&R 风格（左括号不换行）
  ```c
  void Function(void)
  {
      if (condition) {
          do_something();
      }
  }
  ```
- **函数之间**：空 1 行分隔

### 头文件规范
- 使用 `#ifndef` / `#define` / `#endif` 防止重复包含
  ```c
  #ifndef __XXX_H
  #define __XXX_H
  
  // 声明内容
  
  #endif
  ```
- 头文件只放声明、类型定义、宏定义，不放函数实现
- #include 顺序：先自己的头文件，再标准库，再其他模块

### 编程习惯
- 全局变量加 `static`（文件内可见）或 `volatile`（中断共享）
- 函数参数做合法性检查（空指针、越界等）
- 返回值用枚举或明确的状态码，避免歧义
- 关键状态/错误定义成枚举，不用魔数（magic number）

## 第一阶段：W25Q64 读取回顾与应用封装

状态：未开始

目标：复习 W25Q64 的 SPI 读取操作，理解 `bsp_w25q64`（底层驱动层）和 `w25q64`（应用层封装）两层结构的设计理由。

1. 复习 SPI1 硬件初始化
   - PA4=CS, PA5=SCK, PA6=MISO, PA7=MOSI
   - 配置 SPI1 为主机、全双工、8 位数据、CPOL=Low/CPHA=1Edge
   - 分频系数 `SPI_BaudRatePrescaler_4` → 18MHz

2. 封装 SPI 字节交换函数
   - `W25Q64_SPI_SwapByte(tx_data)`：发送一字节并同时接收一字节
   - 等待 TXE → 发送 → 等待 RXNE → 读取

3. `W25Q64_Init` 和读 JEDEC ID
   - 片选拉高释放
   - 命令 `0x9F` + 读回 3 字节（厂商 ID、类型、容量）
   - W25Q64 标准 ID 值为 `0xEF4017`

4. 理解两层封装设计
   - `bsp_w25q64.c`：纯 SPI 协议层（字节交换、读、写、擦除）
   - `w25q64.c`：应用层（`W25Q64_Read()` 带地址越界检查、`W25Q64_Is64MbitCompatible()` 容量判断）
   - 为什么这样设计：底层驱动只做 SPI 通信，应用层负责安全检查，上层（WAV 解析/播放器）不直接操作 SPI

5. 启动项目验证
   - 初始化时钟、NVIC、GPIO、SPI1、W25Q64
   - 读取 JEDEC ID
   - OLED 显示 OK 或 FAIL
   - 编译 → 下载 → 验证

## 第二阶段：WAV 文件头解析

状态：未开始

目标：理解 WAV 文件结构，实现从 W25Q64 读取并解析 WAV 头，提取 PCM 参数。

1. WAV 文件格式入门
   - RIFF 容器：文件以 `RIFF` 标志开头 + 文件大小 + `WAVE` 类型标志
   - fmt chunk：`fmt `（注意后面有空格）+ 长度 + 音频格式/声道数/采样率/位深
   - data chunk：`data` + 长度 + PCM 数据本体
   - chunk 按偶数字节对齐（`chunk_size + (chunk_size & 1)`）

2. 实现小端读取辅助函数
   - `WAV_ReadLe16(data)`：2 字节小端转 uint16_t
   - `WAV_ReadLe32(data)`：4 字节小端转 uint32_t
   - WAV 文件所有多字节数字都是小端格式

3. 实现 WAV_Read 封装
   - 调用 `W25Q64_Read()` 从 Flash 读取小块数据
   - 每次只读 chunk 头，不必读整文件

4. 实现 RIFF 查找
   - 在前 64 字节范围内逐字节查找 `RIFF` 标志
   - 找到后验证后面必须是 `WAVE`
   - 返回 RIFF 实际地址偏移

5. 扫描 chunk 链表
   - 从 RIFF + 12 开始，逐个读取 chunk 头（4 字节 ID + 4 字节长度）
   - 找到 `fmt `：解析音频格式、声道数、采样率、位深等
   - 找到 `data`：记录 PCM 数据偏移和大小
   - 两个 chunk 都找到后停止扫描

6. 参数校验
   - 必须是 PCM 格式（audio_format == 1）
   - 支持 8bit 和 16bit、单声道和双声道
   - 采样率不能为 0，block_align 不能为 0

7. 调试打印
   - 串口输出采样率、位深、声道数、PCM 偏移和大小
   - 确认解析结果正确

## 第三阶段：PWM 音频输出基础

状态：未开始

目标：理解 PWM 发声原理，配置 TIM3_CH3 在 PB0 输出 PWM 波形。

1. PWM 发声原理
   - PWM 脉冲的占空比决定输出电平（经低通滤波后的平均电压）
   - 改变 CCR3 的值 → 改变占空比 → 改变喇叭振膜位置
   - 快速连续改变 CCR3 → 振膜振动 → 产生声音
   - 载波频率（TIM3 的 ARR）决定 PWM 分辨率，本例 ARR=1023（10 位分辨率）

2. 配置 TIM3_CH3 PWM
   - 使能 GPIOB 和 AFIO 时钟
   - PB0 配置为复用推挽输出（`GPIO_Mode_AF_PP`）
   - TIM3：Prescaler=0, Period=1023, 向上计数
   - 通道 3：PWM1 模式，输出使能
   - 初始 CCR3 设为中点（1023/2=511），避免突然偏置

3. 手动改变占空比测试
   - 固定设置 CCR3=100 → 小占空比
   - 固定设置 CCR3=900 → 大占空比
   - 让 CCR3 在 100 和 900 之间循环 → 听到固定频率的"嘀嘀"声

## 第四阶段：采样定时器（TIM2 中断）

状态：未开始

目标：理解采样率概念，配置 TIM2 以 WAV 文件采样率进入中断，在中断中输出样本。

1. 采样率概念
   - 每秒输出的音频样本数量（如 8000Hz、16000Hz、44100Hz）
   - 每个中断输出一个样本到 PWM
   - TIM2 周期计算公式：`period = 72000000 / sample_rate`

2. 配置 TIM2 更新中断
   - Prescaler=0（72MHz 计数时钟）
   - Period 根据采样率计算
   - 开启 TIM2 更新中断（`TIM_IT_Update`）

3. 实现 TIM2_IRQHandler
   - 清除中断标志
   - 从数组中取出下一个样本值
   - 写入 `TIM3->CCR3` → 改变占空比
   - 样本播完后停止

4. 用测试数组验证
   - 准备一个简短的样本数组（例如 512 个值，从 0~1023 变化）
   - TIM2 中断逐个取出写入 CCR3
   - 应该能听到周期性的声音

## 第五阶段：双缓冲播放机制

状态：未开始

目标：从 W25Q64 读取真实的 WAV PCM 数据，用双缓冲机制保证播放连续不断。

1. 为什么需要双缓冲
   - 从 Flash 读取数据需要时间（SPI 读取有等待）
   - 如果在中断里等 Flash 读取，会阻塞中断影响采样时序
   - 方案：两个 PWM 缓冲区，一个在中断中播放，另一个在主循环中填充
   - 播放完一个缓冲区后立刻切换，主循环给空的那个补数据

2. PCM 数据到 PWM 占空比的转换
   - 8bit PCM（0~255 无符号）：左移 8 位变成 0~65535 范围
   - 16bit PCM（-32768~32767 有符号）：先转无符号（+32768），再映射到 0~65535
   - 双声道：左右声道取平均值混为单声道
   - 最后用 `AudioPlayer_U16ToPwm()` 从 0~65535 映射到 0~1023

3. `AudioPlayer_FillBuffer` 实现
   - 从 Flash 读取一段 PCM（最多 512 个采样帧）
   - 遍历每个采样帧，根据位深和声道数计算 PWM 占空比
   - 存入 `pwm_buffers[buffer_id]`
   - 更新 `buffer_ready` 和 `buffer_valid_samples`

4. TIM2 中断消费缓冲区
   - 从 `active_buffer` 中逐样本取出，写入 CCR3
   - 样本播完后切换到另一个缓冲区
   - 如果下一个缓冲区还没准备好 → `underflow` 错误

5. 主循环填充空闲缓冲区
   - `AudioPlayer_Process()` 检查两个缓冲区的状态
   - 对 `buffer_ready[i]==0` 且还有剩余数据的缓冲区调用 FillBuffer
   - 在 main 的 while(1) 中持续调用

6. 完整播放流程串联
   - main：初始化 → WAV 解析 → `AudioPlayer_Init` → `AudioPlayer_Start`（先填满两个缓冲）→ 主循环 Process
   - TIM2 中断：样本输出 → 缓冲区切换 → 播完标记 DONE
   - OLED 显示播放状态，串口输出错误日志
