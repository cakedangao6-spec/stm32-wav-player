# STM32 MP3/WAV 播放器项目硬件说明书

## 1. 设计基线

本硬件设计不是重新发明一个播放器，而是直接承接当前已经完成的软件工程。此前的软件阶段主要验证了 WAV/MP3 的读取、解析和播放逻辑，**并没有形成完整的功放硬件电路**，所以本次硬件落地会把音频输出链路补完整。

- 主控：`STM32F103C8T6`
- 外部存储：`W25Q64`
- 显示：`SSD1306 OLED`，当前代码使用 `PB8/PB9` 软件 I2C
- 串口：`USART1 / PA9 / PA10`
- 当前真实音频输出：`PB0 / TIM3_CH3` PWM
- 当前 Flash 接口：`SPI1 / PA4~PA7`

因此，硬件最终采用以下主方案：

```text
5V输入
 ├─> 3.3V数字电源 -> STM32 / W25Q64 / OLED
 └─> 5V音频电源 -> PAM8403

PB0 PWM
 -> 两级RC低通
 -> 音量调节
 -> PAM8403
 -> 喇叭
```

这套方案的核心原则是：**先与现有固件完全兼容，再为以后升级预留余地。**

## 2. 关键方案选择

### 2.1 音频输出路线比较

| 路线 | 是否适合当前板子 | 说明 |
| --- | --- | --- |
| 片内 DAC | 不适合 | `STM32F103C8T6` 没有片内 DAC |
| PWM + RC | 最适合当前版本 | 与现有代码完全一致，器件少，适合第一版 |
| 外部 SPI DAC | 可升级 | 音质可提升，但会增加器件和软件复杂度 |
| I2S DAC 模块 | 后续升级路线 | 音质更好，但需要更完整的数字音频架构 |

### 2.2 为什么当前板子选 PWM，而不是直接设计 DAC

`STM32F103C8T6` 本身没有片内 DAC，现有代码也已经真实完成了 `TIM3 PWM` 音频输出。若此时硬件强行改成“DAC 版”，就会导致软件和硬件重新脱节，反而不利于项目落地。

当前板子先采用：

```text
PWM -> RC低通 -> 功放
```

优点：

1. 与现有代码直接匹配；
2. 器件少，容易调通；
3. 非常适合学习“数字波形怎样变成模拟音频”。

缺点：

1. 失真和噪声不如真正 DAC；
2. 需要低通滤波；
3. 频响和音质不适合追求高保真。

未来升级时，可以把音频前端换成：

```text
外部DAC / I2S解码器 / 支持DAC的MCU
```

但不建议在当前第一版板子里把这个升级强行混进去。

### 2.3 为什么功放选 PAM8403

最终主方案选 `PAM8403`，原因如下：

1. 5V 供电，能和整板电源体系自然配合；
2. Class-D 效率高，发热小；
3. 器件常见，资料多，成本低；
4. 可直接驱动小功率 4Ω/8Ω 喇叭；
5. 后续如果扩成立体声，也还能继续使用。

与 `LM386` 相比：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| LM386 | 原理直观，适合学习模拟放大 | 效率低，电解电容大，噪声和发热更明显 |
| PAM8403 | 效率高，体积小，适合实际播放器 | 是桥接输出，喇叭不能一端接地；布局更要注意电源噪声 |

### 2.4 喇叭接口怎么定

- 初学者首版推荐：`8Ω / 1W~2W`
- 若追求更大声压，可再评估 `4Ω`
- PAM8403 为桥接输出，喇叭必须接在 `OUT+` 与 `OUT-` 之间，**不能把负端接地**

首版先选 `8Ω` 的原因是：

1. 电流压力更小；
2. 对 5V 供电和 PCB 布局更宽容；
3. 更适合先把系统稳定做出来。

### 2.5 音量控制为什么保留电位器

本设计保留 `RV1 = 10kA` 音量电位器，而不是直接固定增益，原因是：

1. 初次调试时更安全；
2. 可避免功放输入过大导致削顶；
3. 不同喇叭更容易适配；
4. 对学习者更直观。

如果后续产品化、音源幅度和喇叭都固定，也可以改成固定衰减网络。

### 2.6 为什么 OLED 选 I2C，而不是 SPI

当前代码已经使用 `PB8/PB9` 的软件 I2C OLED，显示内容主要是状态、大小、播放提示，并不需要很高刷新率。

I2C 的优点：

1. 只占两根信号线；
2. 方便接现成模块；
3. 与当前软件完全一致；
4. 对当前字符显示已经足够。

SPI OLED 的优点是刷新更快，但要多占 `SCK/MOSI/DC/CS/RES` 等引脚，且当前项目里 SPI1 已经给了 W25Q64。为了少改代码、少增复杂度，本板选择 I2C。

## 3. 模块级原理图设计

### 3.1 STM32 最小系统

#### 电源

- `VDD` 接 `3V3_D`
- `VDDA` 通过 `FB1` 从 `3V3_D` 引入，并在芯片旁放 `0.1uF + 1uF`
- `VBAT` 通过 `0R` 接 `3V3_D`
- 每个 `VDD` 引脚旁边放 `0.1uF` 去耦
- 芯片附近再放 `4.7uF` 作为局部储能

**为什么要放 0.1uF 去耦：**  
MCU 内部门电路翻转时，会在极短时间内需要脉冲电流。长走线和稳压器来不及瞬时供电，离芯片很近的 `0.1uF` 电容可以先把这口电流顶上去，减小电源尖峰和地弹。

**为什么还要放 4.7uF：**  
`0.1uF` 擅长高频，容量太小，面对稍慢一点的负载变化不够用。`4.7uF` 负责中低频能量补充，两者配合更稳。

#### 晶振

- `Y1 = 8MHz`
- `C6/C7 = 22pF`

**为什么是 8MHz：**  
当前固件配置就是 `HSE 8MHz -> PLL x9 -> 72MHz`，换晶振会让时钟树和所有定时逻辑都要重新核算。

**为什么常见用 22pF：**  
晶振需要看到某个负载电容。若晶体标称负载约 `12pF`，两颗相等电容加上 PCB 杂散后通常会落在合适区间。`22pF` 是常见经验值，不是魔法数字；若你以后换了晶体型号，应按其 `CL` 重新计算。

#### 复位

- `R1 = 10k` 从 `NRST` 上拉到 `3.3V`
- `C8 = 0.1uF` 从 `NRST` 到 `GND`
- `SW1` 按键把 `NRST` 拉到 `GND`

**为什么复位上拉常用 10k：**

1. 阻值足够小，能让 NRST 可靠保持高电平；
2. 又不会在按下复位键时产生太大电流；
3. 结合 `0.1uF` 可得到约 `1ms` 的上电延时，简单又常用。

#### BOOT

- `BOOT0` 用 `10k` 下拉，默认从用户 Flash 启动
- `JP1` 允许必要时把 `BOOT0` 拉高
- `PB2/BOOT1` 用 `10k` 下拉

**为什么 BOOT0 不能悬空：**  
悬空输入没有确定电平，可能让 MCU 偶发进入系统 Bootloader，表现就是“有时正常，有时像没烧程序”。

#### SWD

`J2` 至少引出：

- `3V3`
- `SWDIO / PA13`
- `SWCLK / PA14`
- `NRST`
- `GND`

这样既能下载，也能在线调试。对于嵌入式板子，**保留调试口不是可选项，而是生命线。**

### 3.2 W25Q64

连接关系：

| W25Q64 | STM32 |
| --- | --- |
| CS | PA4 |
| CLK | PA5 |
| DO / MISO | PA6 |
| DI / MOSI | PA7 |

#### 为什么这样接

这是 STM32F103 的 `SPI1` 默认引脚，当前软件已经按这一组引脚实现并通过了真实回读验证，所以硬件继续使用最稳妥。

#### 为什么 CS 需要上拉

MCU 上电复位期间，引脚还没完成初始化。若 `CS` 漂浮，Flash 可能误以为自己被选中，MISO 线也可能在不该驱动时驱动。加 `10k` 上拉后，默认就是“未选中”。

#### 为什么 WP# 和 HOLD# 也要上拉

这两个脚如果暂时不用，也不能悬空。上拉后默认：

- 不写保护；
- 不暂停 SPI 通信。

#### 为什么 Flash 一定要就近放去耦

Flash 读写时边沿快、电流变化快。若电源回路太长，可能出现：

- 读写不稳定；
- 高速 SPI 下偶发错位；
- 对周边数字信号造成干扰。

### 3.3 OLED

本设计采用 I2C OLED：

- `PB8 -> SCL`
- `PB9 -> SDA`
- `R9/R10 = 4.7k` 上拉到 3.3V

#### 为什么 4.7k 常见

I2C 是开漏总线，器件只能主动拉低，不能主动推高。上拉电阻负责把线恢复到高电平。阻值太大，上升沿太慢；阻值太小，拉低时电流又会过大。对短距离、100kHz 到 400kHz 的板级 I2C，`4.7k` 是非常常见的折中值。

### 3.4 音频输出

#### 当前固件对应的真实链路

```text
PB0 / TIM3_CH3
 -> R11/C17 一级低通
 -> R12/C18 二级低通
 -> RV1 音量
 -> C19/C20 输入耦合
 -> PAM8403
 -> 喇叭
```

#### 为什么先低通

PWM 不是连续模拟电压，而是高频开关波形。当前代码的 PWM 载波频率约为：

```text
72MHz / 1024 ≈ 70.3kHz
```

真正想保留的是音频包络，不是 70kHz 的载波，所以必须先做低通。

#### 为什么用两级 RC

一阶 RC 低通衰减不够陡。当前采用两级：

- `R = 2.2k`
- `C = 10nF`
- 单级截止频率约 `7.2kHz`

当前验证音频是 `8kHz` 采样，理论有效音频上限约 `4kHz`。因此把滤波截止放在 `7kHz` 左右，可以：

1. 让 0~4kHz 的语音内容基本保留；
2. 又明显压低 `70kHz` PWM 载波；
3. 让学习者在示波器上能明显看到“方波变圆滑”的过程。

#### 为什么要加耦合电容

PWM 低通后的信号会带有直流平均值，约在 `1.65V` 附近。功放输入需要的是交流音频分量，不希望这部分直流跟着进去，所以用输入耦合电容把直流隔开。

`1uF` 配合后级输入阻抗时，低频截止已经很低，对语音播放影响很小。

#### 为什么不能把喇叭直接接 MCU

普通喇叭阻抗低，需要的电流远大于 GPIO 能承受的范围。直接接会导致：

1. 声音很小；
2. 波形严重失真；
3. 可能损伤 MCU 引脚。

#### 为什么 PAM8403 的喇叭端不能接地

PAM8403 是桥接输出。每个喇叭都应接在 `OUT+` 与 `OUT-` 之间，两个端子都在动。若把某一端硬接地，相当于破坏了输出结构，轻则失真，重则损坏功放。

## 4. PCB 设计规划

### 4.1 摆放顺序

1. 先放接口：电源、SWD、UART、OLED、喇叭；
2. 再放主控 U1；
3. 把晶振和最小系统器件贴近 U1；
4. 把 W25Q64 放在 SPI 引脚旁边；
5. 把 OLED 接口放在板边，方便安装显示屏；
6. 把功放和喇叭端子放到另一侧，远离晶振和 Flash；
7. 最后补测试点。

### 4.2 关键布局规则

- 晶振到 MCU 走线要短、对称、少过孔；
- `0.1uF` 去耦必须“贴脚放”，不是“在附近就行”；
- SPI 线短而直，避免绕音频区；
- 功放的 5V 储能电容要靠近功放电源脚；
- 两级 RC 滤波和音量电位器应靠近功放输入侧；
- 低电平音频线不要从时钟线、SWD线和功放输出线旁边长距离平行经过。

### 4.3 模拟地与数字地

这块两层小板不建议把地平面切开。更好的做法是：

1. 保持一个连续完整的 GND 面；
2. 通过器件分区与走线方向控制回流路径；
3. 让功放大电流回路局限在功放附近；
4. 不让喇叭回流从 MCU、晶振、Flash 底下穿过去。

很多初学者一看到“模拟”和“数字”就急着分地，结果反而把回流路径切断，噪声更糟。

### 4.4 初学者常见错误

1. 去耦电容离芯片太远；
2. 晶振走线又长又绕；
3. OLED I2C 忘记上拉；
4. Flash 的 `WP#`、`HOLD#` 悬空；
5. 只看原理图，不看功放电流回路；
6. 把 PAM8403 的负端当成 GND；
7. 让低电平音频线和 PWM/时钟线并排跑很久；
8. 没留 SWD 和测试点，调试时只能“猜”。

## 5. 供调试使用的测试点

| 测试点 | 观察内容 |
| --- | --- |
| TP1 3V3 | 数字电源是否稳定 |
| TP2 GND | 万用表/示波器参考地 |
| TP3 PWM | PB0 是否输出 PWM |
| TP4 FILT | RC 后是否形成模拟音频 |
| TP5 NRST | 复位过程是否正常 |

## 6. 调试顺序

1. 不焊 MCU，先确认 5V 与 3.3V；
2. 焊 MCU 最小系统，确认 `NRST`、晶振、SWD；
3. 烧入最简单闪灯程序；
4. 接 UART，确认日志；
5. 接 W25Q64，先读 JEDEC ID，再做 readback；
6. 接 OLED，确认 I2C 显示；
7. 最后再接音频链路；
8. 先看 `TP3`，再看 `TP4`，最后接功放和喇叭。
