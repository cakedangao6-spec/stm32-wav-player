# 一、项目目标

## 1.1 最终目标

本项目最终目标是完成一个基于 STM32 的 WAV 音频播放系统：

1. 电脑通过串口把 WAV 文件发送给 STM32；
2. STM32 将文件写入外部 SPI Flash `W25Q64`；
3. 播放时，STM32 从 W25Q64 中读取 WAV 文件；
4. 解析 WAV 头，找到 PCM 音频数据；
5. 以正确采样率输出音频数据；
6. 经过 DAC、功放后驱动喇叭播放声音。

## 1.2 系统流程

```text
电脑
 ↓
串口
 ↓
STM32
 ↓
W25Q64
 ↓
WAV解析
 ↓
DAC
 ↓
功放
 ↓
喇叭
```

## 1.3 当前阶段的意义

在继续开发 WAV 播放之前，必须先确认两个最基础的问题：

1. 文件是否真的被完整写入 W25Q64；
2. STM32 从 W25Q64 读出的字节流，是否与电脑上的原始文件逐字节一致。

如果这一步没有做完，后续即使 WAV 解析、定时器和音频输出代码写得正确，最终也可能是在播放一份已经损坏或根本没有写进去的数据。因此，当前阶段先完成“写入后回读验证”，是整个音频项目最重要的地基之一。

# 三、开发流程

## 3.1 起点：先在现有 Keil 工程上继续开发

本次开发没有重新创建无关工程，而是在原教程目录演进出的 Keil5 工程上继续工作；该工程现已正式命名为 `STM32_MP3_WAV播放器`。工程本身已经包含：

- 标准库启动文件和外设库；
- OLED、LED、按键等基础硬件模块；
- W25Q64 底层驱动；
- 文件传输状态机 `app_file_transfer.c`；
- 串口协议层 `protocol.c`；
- 环形缓冲区模块。

当前需求不是立刻做 WAV 播放，而是先确认已经写入 Flash 的 WAV 文件是否完全正确。因此先补齐“只读验证链路”，而不是贸然往播放功能上叠代码。

## 3.2 新增验证链路

为了把验证功能与原有写入流程分开，新增了以下模块：

- `w25q64.c / w25q64.h`
  - 对底层 `bsp_w25q64` 做一层验证侧封装；
  - 提供 JEDEC ID 读取、容量兼容判断、范围检查后的连续读取接口。
- `verify.c / verify.h`
  - 处理 `PING`、`ID?`、`READBACK` 命令；
  - 从 W25Q64 连续读取固定长度数据；
  - 通过串口回传二进制流；
  - 同时在 STM32 侧计算 CRC32。
- `uart_tx.c / uart_tx.h`
  - 提供阻塞式二进制发送接口，保证回读数据可以原样输出。
- `compare.py`
  - 在电脑端自动执行串口握手、接收、保存、比对。

这一步的目标很明确：让“Flash 里到底是什么”变成可测量事实，而不是靠推测。

## 3.3 第一轮失败：Python 等待 `READY` 超时

初版验证程序烧录后，电脑端脚本报错：

```text
错误: 等待 STM32 READY 超时。请确认固件已烧录、串口接线正确、波特率一致。
```

这时还不能直接判断是软件错还是硬件错。因为“收不到 READY”可能由很多原因引起：

- 串口被别的软件占用；
- 串口线接错；
- 波特率不一致；
- 固件没有真正运行；
- 固件运行了但没有收到命令；
- 固件收到命令但内部校验拒绝执行。

因此没有急着改更多业务逻辑，而是先把问题拆开。

## 3.4 先证明硬件链路没有问题

用户提供了一个已知正常的参考工程：

```text
D:\STM32入门教程资料\程序源码\STM32Project-有注释版\9-2 串口发送+接收
```

对比后可以确认，该参考工程使用 USART1，采用 `RXNE` 中断接收。把它烧录到板子上后，使用电脑端串口测试可以正常完成字节回显，因此得到两个结论：

1. USB 转串口、接线、COM4、板子上的 PA9/PA10 通路都没有硬件性故障；
2. 问题应继续在当前工程的软件实现里查。

这一步很关键。它避免了在硬件已经正确的情况下，继续无效地怀疑杜邦线和串口模块。

## 3.5 找到多个叠加问题

继续排查当前工程后，陆续发现以下问题：

### 3.5.1 串口被占用

调试过程中曾发现 `XCOM V2.6` 占用了 `COM4`。当串口助手未关闭时，Python 无法正常独占串口。关闭占用程序后，电脑端脚本才能继续测试。

### 3.5.2 环形缓冲区过大

早期工程里的环形缓冲区曾设置为 `8192` 字节。对 STM32F103C8T6 这种 RAM 很有限的型号来说，这样的静态缓冲区过于激进，导致 RAM 布局进入危险区，固件启动表现不稳定。后续将 `RING_BUF_SIZE` 收敛为 `512` 字节后，内存占用明显回到可控范围。

当前 `Project.map` 中可以看到：

```text
usart1_rx_ring = 520 bytes
__initial_sp   = 0x20000760
```

### 3.5.3 Flash 型号判断过窄

实际硬件读取到的 JEDEC ID 为：

```text
0x00207017
```

而旧逻辑按固定值 `0x00EF4017` 做完全相等判断。这样即使芯片容量同样是 64 Mbit，也会被程序误判为错误器件。后续改为根据容量码 `0x17` 判断 64 Mbit 兼容设备，避免把厂家差异误判为硬件故障。

### 3.5.4 空闲状态下串口输出过多

旧版验证逻辑在空闲时高频输出状态信息，串口被这些文本占用后，电脑端发来的短命令更容易被淹没，导致握手不稳定。后续去掉空闲刷屏，只保留命令驱动式输出，串口协议变得更清晰。

### 3.5.5 接收路径不稳定

参考工程使用的是 `RXNE` 中断接收，而当前工程的验证模式曾一度采用主循环轮询 USART 接收。主循环中还包含 OLED 刷新和状态处理，时序上不如中断接收稳妥。最终将当前工程改为：

```text
USART1 RXNE 中断
    ↓
环形缓冲区
    ↓
上层协议处理
```

同时补上主程序中的时钟初始化调用，使固件行为更完整、更明确。修改后，电脑端能稳定收到：

```text
PONG
```

这说明握手链路真正打通。

## 3.6 第一次完整回读：链路通了，但数据不对

在串口链路修好之后，第一次完整回读终于成功跑完。结果是：

```text
文件大小一致
MD5 不一致
CRC32 不一致
逐字节不一致
第一个错误位置 = 0
不同字节数量 = 283535
```

进一步查看 `readback_before_rewrite.wav` 后发现：

```text
首字节: A1 B2 C3 D4 FF FF FF FF ...
非 FF 字节总数: 4
```

这说明当前 W25Q64 的内容并不是 `666.wav`，而是只有开头 4 字节被写入，其余几乎全是擦除态 `0xFF`。至此可以确定：

1. 回读程序本身已经能正确把 Flash 内容搬回电脑；
2. 真正的问题是 Flash 中原本就没有完整 WAV 文件。

这个结论非常重要。否则很容易误以为“回读比对失败 = 回读程序错误”，从而在错误方向上继续消耗时间。

## 3.7 回头修复写入链路

为了完成真正的闭环验证，不能只证明“读得出来”，还必须把正确文件重新写进去。

工程中原本已经有 `app_file_transfer.c` 写入状态机，但在串口接收架构改为中断方式后，它仍保留旧的 DMA 轮询依赖。这个调用已经不再适合当前接收架构，因此进行了修正，使写入模块重新基于当前环形缓冲数据工作。

随后将主程序临时切换到写入模式，执行：

```text
PING
WRITE:301810
发送 666.wav 全部字节
VERIFY
```

STM32 返回：

```text
PONG
READY
OK:301810
OK:0x0A3A8C9E
```

其中 `0x0A3A8C9E` 与电脑端原始文件 CRC32 一致，说明重新写入后的 Flash 内容已经正确。

## 3.8 切回验证固件并完成最终闭环

写入完成后，主程序又切回验证模式：

```c
#define APP_MODE_VERIFY 1U
```

重新编译、烧录后，再次运行 `compare.py`。最终：

```text
READY:301810
DONE:301810:0x0A3A8C9E
PASS
W25Q64内容完全正确
```

至此，当前阶段真正完成了：

```text
电脑原文件
    ↓
串口写入
    ↓
W25Q64
    ↓
STM32 回读
    ↓
电脑重新保存
    ↓
逐字节完全一致
```

# 四、问题与解决过程

## 4.1 问题一：STM32 `READY` 超时

### 现象

Python 脚本报错：

```text
等待 STM32 READY 超时
```

### 排查

1. 检查是否已烧录最新固件；
2. 检查串口是否被其他软件占用；
3. 检查电脑端与 STM32 端波特率是否一致；
4. 用已知正常的串口参考工程验证硬件链路；
5. 观察 STM32 是否真的能回应命令。

### 原因

该问题不是单一原因造成，而是多个问题叠加：

1. 串口曾被 `XCOM V2.6` 占用；
2. 当前工程接收路径最初不够稳；
3. 程序曾存在空闲状态高频输出；
4. Flash JEDEC ID 的严格匹配逻辑会阻断后续动作。

### 解决

1. 关闭占用 COM4 的串口助手；
2. 改为 USART1 中断接收 + 环形缓冲；
3. 去掉无意义的空闲刷屏；
4. 采用 `PING/PONG` 主动握手；
5. 将 JEDEC 判断改为按 64 Mbit 容量兼容判断。

## 4.2 问题二：串口硬件是否可靠一度无法确定

### 现象

当前工程收不到稳定命令时，很容易怀疑是接线或串口模块坏了。

### 排查

1. 使用用户提供的正常例程进行交叉测试；
2. 发送单字节并检查回显；
3. 确认 `COM4`、PA9、PA10 整条链路工作正常。

### 原因

真正的问题不在硬件，而在当前工程的软件实现。

### 解决

先用独立、已知正确的例程证明硬件无误，再回头集中排查本工程代码。

## 4.3 问题三：波特率与测试效率

### 现象

早期验证阶段使用 `9600 baud`。虽然可以工作，但完整回读 `301810` 字节耗时过长，不利于重复测试。

### 排查

1. 确认固件和 Python 两端波特率必须完全一致；
2. 先在低速下确认基础链路；
3. 链路稳定后提高到 `115200 baud`。

### 原因

`9600 baud` 不是功能性错误，但会显著放大调试成本；同时任何一端忘记同步波特率，都可能表现为乱码或等待超时。

### 解决

当前工程统一改为：

```text
USART1_BAUDRATE = 115200
compare.py 默认波特率 = 115200
```

在此配置下，完整回读耗时约半分钟量级，适合反复验证。

## 4.4 问题四：写入后 Flash 内容不正确

### 现象

第一次完整回读虽然长度正确，但与 `666.wav` 完全不一致：

```text
MD5 不一致
CRC32 不一致
逐字节不一致
第一个错误位置 = 0
```

### 排查

1. 保存回读文件；
2. 查看首字节内容；
3. 统计非 `0xFF` 字节数量；
4. 对比原文件与回读文件摘要。

### 原因

Flash 实际内容为：

```text
A1 B2 C3 D4 FF FF FF ...
```

说明此前 Flash 中并没有完整的 WAV 文件。回读逻辑只是忠实地把错误内容读了出来。

### 解决

1. 修复写入状态机与当前串口接收架构的衔接；
2. 重新通过串口发送 `666.wav`；
3. 使用写入固件内部的 `VERIFY` 命令先做 CRC32 自检；
4. 再切回验证固件做独立回读比对。

## 4.5 问题五：读取与识别逻辑过于死板

### 现象

板子实际读到 JEDEC ID 为：

```text
0x00207017
```

如果继续按固定值 `0x00EF4017` 判断，程序会把当前器件视为错误芯片。

### 排查

1. 通过串口打印 JEDEC ID；
2. 对照当前读到的真实值；
3. 判断真正需要验证的是容量是否符合，而不是厂商码必须完全相同。

### 原因

旧实现把“64 Mbit Flash 可用”错误地等价成“必须是某一个完全固定的 JEDEC ID”。

### 解决

新增：

```c
W25Q64_Is64MbitCompatible()
```

当前按容量码 `0x17` 判断 64 Mbit 兼容芯片，使程序对等价容量器件更稳健。

---

## 4.6 问题六：环形缓冲区设计过大

### 现象

早期大缓冲区配置让固件表现异常，且对 STM32F103C8T6 的 RAM 非常不友好。

### 排查

1. 检查 `ring_buffer.h`；
2. 检查链接后的 RAM 布局；
3. 评估当前任务真实需要的缓存容量。

### 原因

为了“看起来更安全”而盲目增大静态缓冲区，在小 RAM 单片机上可能反而导致系统不稳定。

### 解决

将环形缓冲区调整为：

```c
#define RING_BUF_SIZE 512U
```

在当前命令接收模式下已经足够，同时显著降低 RAM 压力。

---

## 4.7 问题七：文件比对异常

### 现象

第一次回读后，文件大小一致，但摘要和逐字节比较全部失败。

### 排查

1. 不能只看文件大小；
2. 同时检查 MD5；
3. 同时检查 CRC32；
4. 必须做逐字节比较；
5. 查看第一个错误位置和差异总数；
6. 必要时直接查看二进制头部。

### 原因

“长度一致”只能说明读了同样多的字节，不能说明内容正确。第一次回读就是典型例子：长度正确，但几乎整份数据都不对。

### 解决

`compare.py` 现在固定输出：

1. 大小是否一致；
2. MD5 是否一致；
3. CRC32 是否一致；
4. 逐字节是否一致；
5. 不同字节数量；
6. 第一个错误位置。

这样可以在第一时间区分“传输长度问题”和“真实数据内容问题”。

无论这些代码最初来自手写还是 AI 生成，都说明一个原则：AI 给出的嵌入式代码不能直接当成已经验证过的硬件事实，必须经过编译、烧录、串口观测、Flash 回读和真实数据比对。

2. 识别项目为标准库工程，而非 HAL 工程；
3. 新增回读验证模块与电脑端比对脚本；
4. 自动编译并烧录固件；
5. 发现 `READY` 超时后，没有直接猜测，而是先用正常例程确认硬件；
6. 关闭占用 COM4 的串口助手；
7. 缩小环形缓冲区；
8. 放宽 W25Q64 识别逻辑；
9. 去掉空闲刷屏；
10. 将接收链路改成 USART1 中断 + 环形缓冲；
11. 将验证速度从 `9600` 提升到 `115200`；
12. 完成第一次完整回读，确认 Flash 原内容本身不正确；
13. 修复写入链路，重新发送 `666.wav`；
14. 通过写入侧 CRC32 和最终回读侧全量比对，完成闭环验证。

# 五、关键调试经验总结

## 5.1 先验证硬件，再验证协议，最后验证数据

推荐顺序是：

```text
硬件链路是否通
    ↓
命令协议是否通
    ↓
数据内容是否正确
```

如果一开始就直接调业务逻辑，很容易把硬件、协议、数据三种问题混在一起。

## 5.2 先用“已知正确例程”切分问题边界

当串口不通时，用已知正常的独立例程测试，是最省时间的办法。它可以快速回答：

- 是硬件坏了；
- 还是当前工程代码有问题。

## 5.3 不要只相信“能收到字节”

能收到几个字符，不代表整条传输链路可靠。对文件传输来说，还必须验证：

- 命令是否稳定；
- 大文件是否连续；
- 写入后是否能完整读回；
- 内容是否逐字节一致。

## 5.4 文件大小一致，不代表文件正确

本次第一次完整回读已经证明这一点。`301810` 字节的文件完全可以长度正确但内容错误。因此必须至少同时检查：

1. 文件大小；
2. MD5；
3. CRC32；
4. 逐字节比较。

## 5.5 不直接相信 AI 生成代码

AI 可以加速搭架子，但不能替代硬件验证。尤其是嵌入式项目中：

- 一个缓冲区大小；
- 一个中断方式；
- 一个等待时机；
- 一个芯片 ID 判断；

都可能让“看起来合理”的代码在真实板子上失败。

## 5.6 必须做 readback 验证

写入成功提示不等于 Flash 内容真的正确。真正可靠的结论只能来自：

```text
写入
    ↓
重新读取
    ↓
与原始文件比对
```

## 5.7 调试时要保存失败样本

本次保留了：

```text
readback_before_rewrite.wav
```

它证明了第一次失败时 Flash 里真实存储的是 `A1 B2 C3 D4 + FF...`，而不是模糊地说“可能没写好”。失败样本是很有价值的证据。

## 5.8 小内存 MCU 上，静态缓冲区不能随手放大

在 PC 上“多给点缓存”往往没什么成本；在 STM32F103C8T6 上，几 KB 的静态数组就足以改变系统命运。设计要服从真实资源边界。

USART1 配置头文件，当前主要信息：

```c
#define USART1_BAUDRATE 115200U
```

它也是后续排查串口配置时最先应该看的文件之一。

### `app_file_transfer.c`

文件写入状态机，负责：

1. 解析 `PING`、`WRITE:<size>`、`VERIFY`；
2. 擦除目标扇区；
3. 将串口收到的字节按页写入 W25Q64；
4. 写入完成后返回接收字节数；
5. 从 Flash 再次读取并计算 CRC32。

它对应的是“电脑 -> STM32 -> W25Q64”的写入路径。

### `compare.py`

电脑端验证脚本，负责：

1. 自动选择或指定串口；
2. 通过 `PING/PONG` 确认验证固件在线；
3. 发送 `READBACK:<size>`；
4. 接收完整二进制数据并保存为 `readback.wav`；
5. 计算并比较大小、MD5、CRC32；
6. 做逐字节比较，输出差异数量和首个错误位置；
7. 在完全一致时输出：

```text
PASS
W25Q64内容完全正确
```

# 八、下一阶段计划

## 8.1 下一阶段目标

当前“文件能正确写进去，也能完整读回来”已经得到证明。下一阶段可以开始进入真正的 WAV 播放链路：

1. 从 W25Q64 读取 WAV 文件；
2. 解析 WAV 头；
3. 找到 PCM 数据区；
4. 根据采样率安排定时输出；
5. 使用 TIM + DAC + DMA 完成连续音频播放。

## 8.2 建议实现顺序

### 第一步：WAV 头解析

优先实现一个独立的 WAV 解析模块，完成以下检查：

1. 文件开头是否为 `RIFF`；
2. 格式是否为 `WAVE`；
3. 是否存在 `fmt ` 块；
4. 是否存在 `data` 块；
5. 读取声道数、采样率、位深、数据区起始地址和长度；
6. 明确只支持当前计划内的 PCM 格式，先不要过早扩展。

当前 `666.wav` 的头部已经可以看出它是一个标准 WAV 文件，因此下一步应把“能解析”做成独立可测模块，而不是把解析逻辑直接揉进播放函数里。

### 第二步：PCM 数据读取

从 `data` 块开始读取 PCM 字节流。建议先做一个“只读取不播放”的测试模式：

1. 读取前若干采样点；
2. 通过串口打印或输出到调试缓冲；
3. 确认读取地址、字节序和采样宽度都正确。

### 第三步：建立播放时钟

用定时器产生固定采样节拍：

```text
采样率 = 定时器更新频率
```

例如当前 WAV 如果是 8 kHz，则定时器应稳定产生 8 kHz 更新事件。先把时钟跑准，再接 DMA 和音频输出。

### 第四步：DAC 与 DMA 播放

推荐把播放分成双缓冲或半缓冲模式：

1. DMA 负责把 PCM 数据连续搬到音频输出端；
2. CPU 在半传输 / 传输完成中断中补充下一段数据；
3. W25Q64 读取和 DMA 输出解耦，避免播放中断粮。

在正式进入这一阶段前，应先再次确认最终音频输出硬件方案、DAC 路线和板级连接，避免软件已经写完后才发现硬件路径需要调整。

## 8.3 下一阶段验收标准

建议把下一阶段验收拆成四级：

1. 能解析 WAV 头并打印正确参数；
2. 能正确定位 PCM 数据区；
3. 能按采样率稳定输出一段已知波形；
4. 能完整播放 `666.wav`，且没有明显卡顿、重复、变速或杂音。

# 九、给未来自己看的踩坑记录

## 9.1 串口相关

1. 不要忘记串口助手可能占用 COM 口；
2. `COM4` 被占用时，Python 端会表现得像“程序坏了”，其实只是端口没拿到；
3. 别只看“线接着”，要用已知正常工程先证明整条链路；
4. 串口波特率必须两端一致；
5. 当前项目已经统一成 `115200 baud`。

## 9.2 烧录相关

1. 改完代码一定要确认重新编译；
2. 重新编译后还要确认真的重新烧录；
3. 如果表现和源码不一致，第一件事不是猜，而是确认板子里跑的是不是最新固件；
4. Keil 日志是证据，不要只凭感觉。

## 9.3 Flash 相关

1. 写入成功提示不等于数据正确；
2. 一定要 readback；
3. 一定要保留失败样本；
4. 看到大量 `FF` 时，优先怀疑 Flash 仍处于擦除态；
5. 不能只看文件大小，必须看内容；
6. JEDEC ID 不要写得过死，先弄清真正要验证的是容量还是厂家。

## 9.4 代码相关

1. 小 RAM MCU 不要随手开超大静态缓冲区；
2. 接收链路最好先把字节稳稳接住，再让业务逻辑慢慢处理；
3. 业务模式切换要清楚，当前靠 `APP_MODE_VERIFY` 区分；
4. 改了底层接收方式后，记得检查上层是否还残留旧 DMA 逻辑；
5. 不要让调试打印淹没正式协议；
6. AI 生成代码只是一份草稿，不是硬件验证结果。

## 9.5 当前最重要的结论

当前阶段已经不是“理论上能写入”，而是已经真实完成了：

```text
电脑原文件 666.wav
    ↓
写入 W25Q64
    ↓
从 W25Q64 全量读回
    ↓
保存为 readback.wav
    ↓
MD5 / CRC32 / 逐字节完全一致
```

在继续做 WAV 播放前，这个基础已经站稳了。
