# 一、项目整体目标

## 1.1 从“电脑里的文件”到“喇叭里的声音”

一个播放器项目，本质上是在做一件事：

```text
把电脑里的数字文件，变成空气中的声音振动
```

从学习角度看，理想的完整链路可以写成：

```text
电脑
 ↓
串口
 ↓
STM32
 ↓
W25Q64
 ↓
WAV
 ↓
PCM
 ↓
DMA
 ↓
DAC
 ↓
功放
 ↓
喇叭
```

## 1.2 每一步到底在干什么

### 电脑

电脑保存原始音频文件 `666.wav`。  
它还是调试工具，负责把文件发给 STM32，并在验证阶段接收回读数据。

### 串口

串口是电脑和 STM32 之间的数据通道。  
本项目用它做两件事：

1. 把 WAV 文件写入 W25Q64；
2. 把 W25Q64 内容读回电脑，做完整比对。

### STM32

STM32 是整个系统的大脑。  
它负责：

1. 控制 SPI Flash；
2. 解析 WAV 文件；
3. 按正确采样率送出音频样本；
4. 管理缓冲区和播放状态。

### W25Q64

W25Q64 是外部 Flash，相当于播放器的小硬盘。  
STM32 自己片内 Flash 不适合反复存放较大的音频文件，所以把 WAV 文件放在外部 Flash 中。

### WAV

WAV 是一种文件容器。  
它不仅包含声音样本，还包含采样率、位宽、声道数等描述信息。

### PCM

PCM 是真正的音频样本数据。  
WAV 文件前面是“说明书”，后面的 PCM 才是“每一个时刻声音有多大”。

### DMA

DMA 是一种自动搬运数据的硬件机制。  
理想情况下，它可以把内存中的音频样本自动送到输出外设，减少 CPU 负担。

### DAC

DAC 是数模转换器。  
数字 0、1、2、3 只是数字，喇叭需要连续变化的电压。DAC 的作用就是把数字样本变成模拟电压。

### 功放

MCU 输出能力很弱，带不动普通喇叭。  
功放负责把小信号放大到足以驱动喇叭。

### 喇叭

喇叭把电信号变成振动，最后形成我们听到的声音。

## 1.3 当前工程的真实实现

当前工程使用的是 `STM32F103C8T6`。  
这份真实代码最终实现的是：

```text
电脑
 ↓
串口
 ↓
STM32
 ↓
W25Q64
 ↓
WAV
 ↓
PCM
 ↓
TIM2 采样节拍
 ↓
TIM3 PWM
 ↓
RC 低通
 ↓
功放
 ↓
喇叭
```

也就是说：

- 当前工程没有实际使用 DAC；
- 当前工程没有实际使用 DMA 输出音频；
- 当前声音输出由 `TIM3 PWM` 完成；
- 当前采样节拍由 `TIM2` 中断完成；
- 当前音频输出引脚是 `PB0`；
- `PA4` 在当前工程中用于 `W25Q64_CS`。

这不是“退而求其次的假实现”，而是针对当前硬件平台做出的真实工程取舍。

## 2.3 谁调用谁

### 播放模式

1. `main()` 调用 `WAV_ParseFromFlash()`；
2. `WAV_ParseFromFlash()` 调用 `W25Q64_Read()`；
3. `W25Q64_Read()` 调用底层 `W25Q64_ReadData()`；
4. `main()` 调用 `AudioPlayer_Init()`；
5. `main()` 调用 `AudioPlayer_Start()`；
6. 主循环持续调用 `AudioPlayer_Process()`；
7. `TIM2_IRQHandler()` 周期性输出样本。

### 写入模式

1. `main()` 进入 `App_FileTransfer_Process()`；
2. 串口字节进入 `usart1_rx_ring`；
3. `Protocol_FeedByte()` 解析命令；
4. `W25Q64_SectorErase()` 擦除扇区；
5. `W25Q64_PageProgram()` 按页写入。

### 验证模式

1. `compare.py` 发送 `PING` 和 `READBACK`；
2. `Verify_Process()` 解析命令；
3. `Verify_DumpFlash()` 连续读取 Flash；
4. 数据经 `UART_TX_SendBuffer()` 回传；
5. 电脑端做 MD5、CRC32、逐字节比较。

# 三、项目开发全过程

## 3.1 第一阶段：先让文件能进 Flash

项目最初先解决“电脑文件如何进入 STM32”这个问题。  
于是先实现了：

1. USART1 通信；
2. 串口协议；
3. W25Q64 擦除；
4. W25Q64 页写入；
5. 写入完成后的 CRC32 自检。

这一步的核心思想是：  
先不要急着播放，先确保文件能被可靠保存。

## 3.2 第二阶段：发现“写进去”不能只靠感觉

仅凭“程序返回 OK”还不够。  
真正可靠的做法是把 Flash 内容再读回来，和电脑原文件逐字节比较。

于是增加了：

- `verify.c / verify.h`
- `uart_tx.c / uart_tx.h`
- 电脑端 `compare.py`

`compare.py` 不是装饰品，而是整个项目的质量门槛。

## 3.3 第三阶段：串口超时问题

### 现象

Python 等待：

```text
READY
```

但一直超时。

### 排查顺序

1. 检查串口是否被占用；
2. 检查是否烧录了最新固件；
3. 检查电脑和 STM32 的波特率；
4. 用一个已知正常的串口工程交叉验证硬件；
5. 最后再回到本工程看软件。

### 真正发现的问题

项目中实际遇到过：

1. `XCOM V2.6` 占用了 `COM4`；
2. 早期接收路径不够稳；
3. 高速空闲打印会干扰命令交互；
4. Flash ID 判断过于死板。

### 解决方法

1. 关闭占用 COM4 的程序；
2. 改成 `USART1 RXNE` 中断接收；
3. 用 `ring_buffer.c` 缓冲接收字节；
4. 去掉空闲刷屏；
5. 用 `PING/PONG` 建立主动握手。

## 3.4 第四阶段：波特率优化

最初验证用过 `9600 baud`。  
这不是逻辑错误，但回读 30 万字节会非常慢。

后来统一调整为：

```text
USART1_BAUDRATE = 115200
compare.py 默认波特率 = 115200
```

工程经验是：  
先在保守速度下确认链路，再提高速度做完整测试。

## 3.5 第五阶段：第一次回读失败

第一次完整回读虽然成功收到了 `301810` 字节，但比对结果失败：

```text
MD5 不一致
CRC32 不一致
逐字节不一致
第一个错误位置 = 0
```

进一步查看失败样本后发现：

```text
A1 B2 C3 D4 FF FF FF ...
```

这说明：

1. 回读程序本身已经能工作；
2. Flash 里面原来并没有完整的 `666.wav`；
3. “长度正确”绝不代表“内容正确”。

## 3.6 第六阶段：修复写入链路

写入状态机 `app_file_transfer.c` 原来还残留旧的 DMA 轮询思路。  
当前工程后来改成了中断接收，所以写入层也要跟着统一。

修正后，重新写入 `666.wav`，STM32 自检返回：

```text
OK:301810
OK:0x0A3A8C9E
```

电脑端回读后也得到：

```text
PASS
W25Q64内容完全正确
```

## 3.7 第七阶段：从存储系统升级为播放器

当 Flash 内容确认可靠之后，才开始做播放功能：

1. 新增 `wav.c / wav.h`；
2. 新增 `audio_player.c / audio_player.h`；
3. `main.c` 增加播放模式；
4. 解析 `666.wav`；
5. 按 `8000 Hz` 输出样本；
6. 使用 PWM 实际播放声音。

最终串口输出：

```text
WAV format: PCM
Sample rate: 8000 Hz
Bits: 16
Channels: 1
PCM offset: 44
PCM size: 301766
Playback start
Playback end
```

实测播放时间约 `18.857 s`，与文件时长相符。

## 3.8 关于 AI 代码的经验

本工程最有价值的教训之一是：

```text
AI 能生成代码，但不能替代硬件验证
```

最容易被 AI 写错或写得过度乐观的地方包括：

1. 误判芯片外设能力；
2. 忽略 RAM 容量；
3. 把某个厂商 ID 写死；
4. 忽略串口占用；
5. 只看“返回 OK”，不做 readback；
6. 忽略当前工程已有引脚占用。

# 四、文件结构

## 4.1 `main.c`

### 作用

`main.c` 是程序入口。  
它负责初始化系统，并根据 `APP_MODE` 进入三种模式之一：

```c
APP_MODE_TRANSFER
APP_MODE_VERIFY
APP_MODE_PLAYBACK
```

当前状态是：

```c
#define APP_MODE APP_MODE_PLAYBACK
```

### 输入

- 固件编译配置；
- W25Q64 中的文件；
- 外部硬件状态。

### 输出

- 串口日志；
- OLED 状态；
- 播放模式下的音频输出。

### 谁调用它

复位后由启动文件进入 `main()`。

### 为什么要独立

入口层只做“组织工作”，不要把 Flash、WAV、播放细节全部揉进去。  
这样当功能变多时，工程仍然能看懂。

## 4.2 `w25q64.c`

### 作用

它是上层使用的包装层。  
底层 SPI 细节在 `bsp_w25q64.c`，上层通过 `W25Q64_Read()`、`W25Q64_ReadJedecID()` 等接口访问。

### 输入

- 读地址；
- 缓冲区；
- 读取长度。

### 输出

- Flash 数据；
- 设备 ID；
- 读操作是否越界。

### 谁调用

- `wav.c`
- `verify.c`
- `main.c`
- `audio_player.c`

### 为什么拆模块

这样上层不用关心 SPI 交换字节、片选拉高拉低等细节。  
它只关心“从某个地址读若干字节”。

## 4.3 `w25q64.h`

### 作用

声明 W25Q64 上层接口和容量相关常量。

### 关键内容

```c
W25Q64_CAPACITY_BYTES
W25Q64_64MBIT_CAPACITY_CODE
W25Q64_Read()
W25Q64_Is64MbitCompatible()
```

### 为什么重要

它把“可公开使用的接口”集中写清楚。  
读 `.h` 文件时，应该先理解模块能提供什么能力。

## 4.4 `wav.c`

### 作用

从 Flash 中解析 WAV 文件结构。

### 输入

- 文件基地址；
- 文件总长度；
- `WavInfo_t` 输出结构体。

### 输出

`WavInfo_t` 中的：

```c
sample_rate
bits_per_sample
channels
pcm_offset
pcm_size
```

### 谁调用

`main.c` 调用 `WAV_ParseFromFlash()`。

### 为什么拆模块

WAV 解析属于“文件格式”问题，不属于“播放输出”问题。  
把它单独拆开，未来无论改成 PWM、DAC 还是 I2S，WAV 解析都可以复用。

## 4.5 `wav.h`

### 作用

定义：

- `WavStatus_t`
- `WavInfo_t`
- `WAV_ParseFromFlash()`
- `WAV_StatusString()`

### 为什么重要

它是文件格式层和上层应用之间的合同。

## 4.6 `audio_player.c`

### 作用

它是当前播放功能的核心。

它负责：

1. 根据 WAV 信息读取 PCM；
2. 把 8-bit 或 16-bit PCM 转成 PWM 占空比；
3. 使用 `pwm_buffers[2][512]` 做双缓冲；
4. 使用 `TIM2` 产生采样节拍；
5. 使用 `TIM3_CH3 / PB0` 输出 PWM；
6. 在 `TIM2_IRQHandler()` 中逐样本播放。

### 输入

- `WavInfo_t`
- W25Q64 中的 PCM 数据。

### 输出

- `PB0` 上的 PWM 波形；
- 播放状态；
- 错误状态。

### 谁调用

- `main.c` 调用 `AudioPlayer_Init()`、`AudioPlayer_Start()`、`AudioPlayer_Process()`；
- 中断向量调用 `TIM2_IRQHandler()`。

### 为什么拆模块

播放逻辑最复杂。  
把它独立出来以后，未来若改成 DAC 版，只需要替换后端，不必重写 WAV 解析和 Flash 层。

## 4.7 `audio_player.h`

### 作用

定义播放器状态、错误码和公开接口。

### 关键内容

```c
AudioPlayer_Init()
AudioPlayer_Start()
AudioPlayer_Process()
AudioPlayer_GetState()
AudioPlayer_GetLastError()
```

## 4.8 `compare.py`

### 作用

它是电脑端的验证工具。

### 输入

- 串口名；
- 波特率；
- 原始文件 `666.wav`；
- STM32 回传的数据。

### 输出

- `readback.wav`
- 文件大小比较；
- MD5；
- CRC32；
- 逐字节比较；
- 差异数量；
- 第一个错误位置。

### 谁调用

由开发者在电脑命令行中手动运行。

### 为什么要单独存在

嵌入式开发不能只靠“看起来能跑”。  
`compare.py` 把“Flash 内容是否真的正确”变成了可重复验证的结论。

# 五、重点原理

## 5.1 SPI 如何读取 W25Q64

SPI 是一种同步串行总线。  
本项目中：

- `PA5` 是时钟；
- `PA6` 是 MISO；
- `PA7` 是 MOSI；
- `PA4` 是片选 CS。

读取时，STM32 会：

1. 拉低 CS；
2. 发送读取命令 `0x03`；
3. 发送 24 位地址；
4. 连续发送空字节 `0xFF`；
5. W25Q64 在 MISO 上返回数据；
6. 读完后拉高 CS。

底层函数是：

```c
W25Q64_ReadData()
```

## 5.2 为什么 Flash 写入前要擦除

Flash 的一个重要特点是：

- 擦除后通常是 `0xFF`；
- 写入时可以把位从 `1` 改成 `0`；
- 但不能随便把 `0` 再直接改回 `1`。

所以如果一块区域以前写过数据，再写新文件前，必须先擦除。

## 5.3 Page 和 Sector 是什么

### Page

W25Q64 中一个页大小为：

```c
W25Q64_PAGE_SIZE = 256
```

页是最小的常用编程单位。  
`W25Q64_PageProgram()` 每次最多写 256 字节。

### Sector

一个扇区大小为：

```c
W25Q64_SECTOR_SIZE = 4096
```

扇区是常见的最小擦除单位。  
写文件前，代码会先按扇区擦除，再按页写入。

## 5.4 WAV 文件结构

WAV 不是单纯的裸音频流。  
它至少包含：

```text
RIFF
WAVE
fmt 
data
```

### `RIFF`

说明这是 RIFF 容器。

### `WAVE`

说明这是 WAV 类型的数据。

### `fmt `

描述声音格式，例如：

- PCM 还是压缩格式；
- 采样率；
- 位宽；
- 声道数。

### `data`

真正的 PCM 样本数据。

## 5.5 为什么常说 WAV 头是 44 字节

最常见的 PCM WAV 文件，头部通常是 44 字节。  
本项目中的 `666.wav` 恰好也是：

```text
PCM offset = 44
```

但要注意，WAV 可能带额外块。  
所以 `wav.c` 没有死背“第 44 字节一定是数据”，而是逐块搜索 `fmt ` 和 `data`。

这就是 `WAV_ParseFromFlash()` 比“直接从 44 开始播”更可靠的原因。

## 5.6 PCM 是什么

PCM 可以理解成“每个采样时刻的音量数字”。

例如当前文件：

```text
8000 Hz
16 bit
1 channel
```

意思是：

1. 每秒有 8000 个采样点；
2. 每个采样点用 16 bit 表示；
3. 只有一个声道。

如果按正确速度把这些点依次送出来，就能还原声音。

## 5.7 为什么 TIM 控制采样率

如果应该每秒播放 8000 个采样点，但你实际只播放 4000 个，声音就会变慢、变低。  
如果你播放 16000 个，声音就会变快、变尖。

所以播放器必须有一个“节拍器”。

当前工程中：

```c
AudioPlayer_ConfigSampleTimer()
```

根据 `wav.sample_rate` 配置 `TIM2`。  
对于 `8000 Hz` 的 WAV，`TIM2` 就以约每秒 8000 次的速度触发 `TIM2_IRQHandler()`。

## 5.8 DAC 是什么

DAC 的全名是数模转换器。  
它把一个数字样本转换成对应的模拟电压。

例如：

```text
数字小 -> 输出电压低
数字大 -> 输出电压高
```

如果按音频采样率连续改变输出电压，经过功放后就能推动喇叭。

### 当前工程中的真实情况

当前代码没有使用 DAC。  
当前方案是把 PCM 转成 PWM 占空比，再由 RC 低通得到近似模拟波形。

## 5.9 DMA 为什么能减少卡顿

DMA 可以在外设和内存之间自动搬运数据。  
如果输出音频时由 DMA 自动把样本送给 DAC，CPU 就不用每个采样点都亲自干预。

它的好处是：

1. CPU 更空闲；
2. 输出更稳定；
3. 更适合高采样率或复杂界面。

### 当前工程中的真实情况

当前播放代码没有使用 DMA。  
`TIM2_IRQHandler()` 每到一个采样点，就手动把一个占空比写入：

```c
TIM3->CCR3
```

这对当前 `8000 Hz`、单声道文件已经足够。  
若未来升级为更高采样率或 DAC 平台，DMA 会更值得引入。

## 5.10 双缓冲为什么存在

如果播放器只用一个缓冲区，会遇到冲突：

1. 输出端正在读它；
2. CPU 又想往里面写新数据。

双缓冲解决方法是：

```text
缓冲 A 正在播放
缓冲 B 同时补充
然后交换
```

当前工程中对应变量是：

```c
pwm_buffers[2][AUDIO_BUFFER_SAMPLES]
buffer_ready[2]
buffer_valid_samples[2]
active_buffer
```

这就是 `AudioPlayer_Process()` 和 `TIM2_IRQHandler()` 能配合工作的原因。

# 七、关键代码讲解

## 7.1 `WAV_ParseFromFlash()`

### 为什么这样写

它没有假设数据一定从第 44 字节开始，而是：

1. 先找 `RIFF`；
2. 再确认 `WAVE`；
3. 循环查找 `fmt ` 和 `data`。

### 如果删掉会怎样

播放器将不知道：

- 采样率是多少；
- 位宽是多少；
- 声道数是多少；
- PCM 从哪里开始。

结果就是无法正确播放，甚至会把文件头当声音播出去。

### 怎么调试

看串口：

```text
WAV format: PCM
Sample rate: ...
Bits: ...
Channels: ...
PCM offset: ...
PCM size: ...
```

## 7.2 `W25Q64_Read()`

### 为什么这样写

它在上层读 Flash 前先做：

1. 空指针判断；
2. 长度判断；
3. 边界判断。

### 如果删掉会怎样

上层一旦传入错误地址，可能读到非法区域，错误会更难查。

### 怎么调试

先看：

```c
W25Q64_ReadJedecID()
```

再看 `verify.c` 的回读结果是否通过。

## 7.3 `AudioPlayer_FillBuffer()`

### 为什么这样写

这个函数做了三件事：

1. 从 W25Q64 取 PCM；
2. 根据 8-bit / 16-bit、单声道 / 双声道转换；
3. 把结果存入 PWM 缓冲区。

### 如果删掉会怎样

中断里没有可播样本，最终会触发：

```c
AUDIO_PLAYER_ERR_UNDERFLOW
```

### 怎么调试

关注：

```c
buffer_ready[]
buffer_valid_samples[]
source_frames_remaining
```

## 7.4 `AudioPlayer_ConfigSampleTimer()`

### 为什么这样写

播放器必须按 WAV 的 `sample_rate` 输出，而不是“主循环跑多快就播多快”。

### 如果删掉会怎样

声音速度会失控，或者根本没有稳定输出。

### 怎么调试

查看串口中的采样率，再检查 `TIM2_IRQHandler()` 是否持续进入。

## 7.5 `TIM2_IRQHandler()`

### 为什么这样写

它是当前工程真正的“逐点播放器”：

1. 每次中断取一个样本；
2. 写入 `TIM3->CCR3`；
3. 更新 `samples_played`；
4. 缓冲用完时切换到另一块。

### 如果删掉会怎样

不会有连续音频输出。

### 怎么调试

可以观察：

```c
samples_played
player_state
last_error
```

如果没有中断，`samples_played` 不会增长。

## 7.6 `AudioPlayer_Process()`

### 为什么这样写

中断只负责“准时输出”，主循环负责“慢一点的补货”。  
这是实时系统里常见的分工。

### 如果删掉会怎样

第一个或第二个缓冲播完后，后续没有新数据，最终 underflow。

### 怎么调试

如果出现：

```text
Audio error: buffer underflow
```

优先检查这个函数是否还在主循环中被调用。

# 八、硬件接线

## 8.1 当前真实接线

### W25Q64

| 功能 | STM32 引脚 |
| --- | --- |
| CS | `PA4` |
| SCK | `PA5` |
| MISO | `PA6` |
| MOSI | `PA7` |

### 音频输出

| 功能 | STM32 引脚 |
| --- | --- |
| PWM 音频输出 | `PB0 / TIM3_CH3` |

## 8.2 功放与喇叭

推荐：

```text
PB0 -> RC 低通 -> 功放输入
STM32 GND -> 功放 GND
功放输出 -> 喇叭
```

## 8.3 PA4 要怎么接

当前真实工程里：

```text
PA4 = W25Q64_CS
```

所以：

1. `PA4` 接 W25Q64 的 CS；
2. `PA4` 不接功放；
3. 音频输入接 `PB0`。

## 8.4 为什么不能直接接喇叭

普通喇叭需要的电流比 MCU 引脚能提供的大得多。  
如果直接接：

1. 声音会很小；
2. 波形会失真；
3. 还可能让 MCU 引脚承受不合适的负载。

功放不是可有可无的装饰，而是把小信号变成能驱动喇叭的功率信号。

# 九、验证方法

## 9.1 怎么看 W25Q64 是否正常

### 方法一：看设备 ID

启动时主程序会读取 JEDEC ID。  
`W25Q64_Is64MbitCompatible()` 用容量码判断设备是否可用。

### 方法二：做 readback

最可靠的方法仍然是：

```text
写入
 ↓
回读
 ↓
compare.py 比对
```

## 9.2 怎么看当前工程有没有播放

当前代码不使用 DAC 和 DMA，所以要按当前实现验证：

1. 串口是否打印 `Playback start`；
2. `PB0` 是否有 PWM 波形；
3. `TIM2_IRQHandler()` 是否运行；
4. `samples_played` 是否增长；
5. 最后是否打印 `Playback end`。

## 9.3 怎么看 TIM 是否正常

### `TIM2`

负责采样节拍。  
如果它不工作：

- `TIM2_IRQHandler()` 不进入；
- `samples_played` 不增长；
- 不会正常结束播放。

### `TIM3`

负责 PWM 输出。  
如果它不工作：

- PB0 没有 PWM；
- 功放输入端没有有效音频。

## 9.4 怎么看 DAC 是否正常

当前工程未使用 DAC。  
这一项属于未来 DAC 版的验证内容。

如果未来改为 DAC 版本，可验证：

1. DAC 输出脚是否有随样本变化的电压；
2. 触发定时器是否运行；
3. DMA 是否持续搬运；
4. 半传输与全传输中断是否按预期发生。

## 9.5 怎么看 DMA 是否正常

当前工程未使用 DMA。  
当前项目中如果你搜索代码，会看到没有播放侧 DMA 配置。

如果未来升级：

1. 检查 DMA 通道使能；
2. 检查传输计数是否递减；
3. 检查 DMA 中断是否触发；
4. 检查输出端是否持续更新。

# 十、踩坑总结

## 10.1 最容易忘记的事情

1. `compare.py` 在工作区的 `工具\串口验证` 目录；
2. 当前串口默认 `115200 baud`；
3. `PA4` 是 W25Q64 片选，不是音频；
4. 当前播放输出脚是 `PB0`；
5. 当前实现是 PWM，不是 DAC；
6. 播放模式下 `APP_MODE` 必须是 `APP_MODE_PLAYBACK`；
7. 文件大小一致不代表内容一致；
8. Flash 写入前必须擦除；
9. 串口助手可能占用 COM 口；
10. 改完代码还要确认真的重新烧录。

## 10.2 AI 最容易写错的地方

1. 把不存在的外设当成一定存在；
2. 忽略已有引脚用途；
3. 随手开过大的静态缓冲区；
4. 把某个固定 JEDEC ID 写死；
5. 只做发送，不做回读；
6. 用主循环硬扛实时任务；
7. 看到“能编译”就误以为“硬件一定能跑”。

## 10.3 如果重新做一次，建议顺序

1. 先列出芯片真实资源；
2. 先验证串口；
3. 再验证 SPI 和 Flash；
4. 先写入一个小测试数据；
5. 再做整文件写入；
6. 做 readback 全量验证；
7. 再解析 WAV；
8. 再做播放；
9. 最后才考虑界面和高级功能。

## 11.3 歌词

可以考虑：

1. 单独保存时间戳歌词文件；
2. 播放时按采样计数换算当前时间；
3. OLED 同步显示歌词。

## 11.4 蓝牙

可以把串口数据源从电脑替换为蓝牙模块。  
但要先明确蓝牙只负责控制，还是也要负责传文件。

## 11.5 MP3 解码

MP3 不是 PCM，不能像 WAV 一样直接按样本播放。  
要支持 MP3，需要增加解码器，把压缩数据先还原成 PCM。

这会显著提高：

1. 算法复杂度；
2. RAM 需求；
3. CPU 负载。

## 11.6 DAC + DMA 版

如果未来更换为支持 DAC 的平台，可以保留：

- `w25q64.c`
- `wav.c`
- 大部分上层流程

然后把 `audio_player.c` 的输出后端改成：

```text
TIM 触发
 ↓
DMA 搬运
 ↓
DAC 输出
```

这会是从“当前能跑”到“更标准音频架构”的自然升级。
